# Technical SEO Audit: webseite.hamburg
# Date: 2026-03-19
# Auditor: Automated Technical SEO Analysis

---

## EXECUTIVE SUMMARY

**Overall Technical SEO Score: 38/100**

The site has a fundamental architectural problem that undermines nearly all SEO efforts:
it is a pure Client-Side Rendered (CSR) React Single Page Application with NO server-side
rendering or pre-rendering. Every URL returns the identical HTML shell (6,663 bytes) with
an empty `<div id="root"></div>`. All content, meta tags, titles, canonical tags, and
structured data for individual pages are rendered exclusively via JavaScript.

While Googlebot can render JavaScript, this architecture creates significant risks and
confirmed issues across crawlability, indexability, and performance.

---

## CRITICAL FINDINGS (Severity: CRITICAL)

### C1. Pure Client-Side Rendering -- No SSR/SSG
- **Status**: FAIL
- **Evidence**: Every URL (/, /leistungen, /leistungen/seo, /wissen/lokales-seo-hamburg-guide,
  /kontakt, /ueber-uns, /nonexistent-page) returns byte-for-byte identical HTML:
  - Same etag: "f014e77566927c720f998c2d1590b9e5"
  - Same content-length: 6,663 bytes
  - Same `<body><div id="root"></div></body>`
- **Impact**: Search engines that do not execute JavaScript (Bing, Yandex, many AI crawlers)
  will see NO content whatsoever. Even Google requires a two-phase indexing process (crawl
  then render), which delays indexing and can fail for some pages.
- **Recommendation**: Migrate to a framework with SSR/SSG support (Next.js, Astro, Remix).
  At minimum, implement pre-rendering/dynamic rendering for crawlers as an interim solution.

### C2. All Pages Share Identical Meta Tags, Title, and Canonical
- **Status**: FAIL
- **Evidence**: The server-delivered HTML for ALL URLs contains:
  - `<title>Webdesign & SEO Hamburg | Websites die Kunden bringen</title>` (homepage title)
  - `<meta name="description" content="Professionelle Websites, SEO und Google Ads...">` (homepage description)
  - `<link rel="canonical" href="https://webseite.hamburg/" />` (homepage canonical!)
  - `<meta property="og:title" content="Webdesign & SEO Hamburg | Websites die Kunden bringen" />`
- **Impact**: The canonical tag on EVERY page points to the homepage. If Google uses the
  server-rendered HTML canonical (before JS execution), it will treat all 16 pages as
  duplicates of the homepage. The JS does update document.title via useEffect, but
  canonical and meta description updates are not confirmed in server HTML.
- **Recommendation**: This is the single most damaging issue. Implement server-side unique
  titles, descriptions, and canonical URLs per page. Each page must have its own canonical
  pointing to itself.

### C3. No Proper 404 Handling
- **Status**: FAIL
- **Evidence**: Requesting /nonexistent-page returns HTTP 200 with the same SPA shell.
  The server never returns a 404 status code for non-existent pages.
- **Impact**: Search engines will index garbage URLs as valid pages. Crawl budget is wasted.
  Soft 404 detection by Google is unreliable.
- **Recommendation**: Configure Vercel rewrites to return 404 status codes for non-existent
  routes. Alternatively, implement server-side route matching.

## HIGH SEVERITY FINDINGS

### H1. Structured Data Only in JavaScript
- **Status**: FAIL
- **Evidence**: The HTML shell contains structured data only for the homepage
  (ProfessionalService + FAQPage). Page-specific structured data (Service, Article,
  AboutPage, BreadcrumbList, CollectionPage, WebPage) is injected via JS using
  `dangerouslySetInnerHTML`.
- **Impact**: Non-JS crawlers cannot parse page-specific structured data. Google may
  parse it after JS rendering, but with delays and potential failures.
- **Schema types found in JS bundle**: Article, AboutPage, BreadcrumbList, CollectionPage,
  FAQPage, ListItem, Offer, Person, ProfessionalService, Question, Service, WebPage.
  This is a good variety -- the data just needs to be server-rendered.

### H2. Large Uncompressed JS Bundle
- **Status**: WARNING
- **Evidence**:
  - JS bundle: 469,462 bytes (459 KB) uncompressed / ~150 KB Brotli compressed
  - CSS bundle: 36,958 bytes (36 KB) uncompressed
  - Contains full React runtime + all page content in one monolithic bundle
- **Impact**: The entire site's content and logic is in a single JS file. This means:
  - LCP is blocked until ~150 KB of JS is downloaded and parsed
  - No code splitting per route
  - All 16 pages' content is loaded even when visiting one page
- **Recommendation**: Implement route-based code splitting with React.lazy() or migrate
  to a framework that handles this automatically (Next.js, Remix).

### H3. No og:image Meta Tag
- **Status**: FAIL
- **Evidence**: No og:image tag found in the HTML source for any page.
- **Impact**: Social media shares (LinkedIn, Facebook, Twitter/X, WhatsApp) will show
  no preview image, significantly reducing click-through rates.
- **Recommendation**: Add og:image (minimum 1200x630px) and og:image:alt tags. Create
  page-specific images for key service pages and articles.

### H4. Missing og:url Tag
- **Status**: FAIL
- **Evidence**: No og:url tag found. The og:type is set to "website" for all pages
  (should be "article" for blog posts).
- **Recommendation**: Add og:url per page. Use og:type "article" for /wissen/* pages
  with article:published_time and article:author tags.

### H5. Static Assets Not Cached Long-Term
- **Status**: WARNING
- **Evidence**: JS and CSS bundles use content-hash filenames (index-CLgbmDEF.js) which
  is good, but cache-control is `public, max-age=0, must-revalidate` -- forcing
  revalidation on every request.
- **Impact**: Repeat visitors must revalidate assets even though the hash ensures
  immutability. This adds unnecessary latency.
- **Recommendation**: Set `Cache-Control: public, max-age=31536000, immutable` for
  hashed asset files in vercel.json.

## MEDIUM SEVERITY FINDINGS

### M1. Google Fonts Render-Blocking
- **Status**: WARNING
- **Evidence**: Two Google Fonts loaded (Goldman, Inter) with 4 weight variants via
  render-blocking `<link>` tags. While preconnect hints are present, the fonts still
  block rendering.
- **Impact**: Adds 100-300ms to LCP depending on connection speed. Potential CLS if
  fonts load after initial paint.
- **Recommendation**: Add `font-display: swap` (already handled by `&display=swap`
  in the URL -- good). Consider self-hosting fonts to eliminate the DNS+connect overhead
  to fonts.googleapis.com and fonts.gstatic.com.

### M2. Impressum and Datenschutz Missing from Sitemap
- **Status**: WARNING
- **Evidence**: Routes /impressum and /datenschutz exist in the JS bundle but are NOT
  listed in sitemap.xml.
- **Impact**: Legal pages for German websites (Impressum is legally required under TMG/DDG)
  are not signaled to search engines. While not critical for rankings, it is a completeness
  issue.
- **Recommendation**: Add /impressum and /datenschutz to sitemap.xml.
  Sitemap currently has 16 URLs; should have 18.

### M3. Sitemap Missing lastmod on Key Pages
- **Status**: WARNING
- **Evidence**: Homepage, /leistungen, all /leistungen/* sub-pages, /ueber-uns, /kontakt,
  and /wissen have no `<lastmod>` date. Only /wissen/* articles have lastmod.
- **Impact**: Google cannot prioritize recrawling of recently updated service pages.
- **Recommendation**: Add accurate lastmod dates to all sitemap entries.

### M4. No Twitter/X Card Meta Tags
- **Status**: FAIL
- **Evidence**: No twitter:card, twitter:title, twitter:description, or twitter:image
  tags found.
- **Recommendation**: Add twitter:card="summary_large_image" and corresponding tags.

### M5. Keywords Meta Tag (Obsolete)
- **Status**: INFO
- **Evidence**: `<meta name="keywords" content="Webdesign Hamburg, SEO Hamburg...">` present.
- **Impact**: Google has ignored the keywords meta tag since 2009. Not harmful, but
  provides no SEO value and exposes keyword strategy to competitors.
- **Recommendation**: Remove or keep -- no SEO impact either way.

### L1. No X-Robots-Tag HTTP Header
- **Status**: PASS (neutral)
- **Evidence**: No X-Robots-Tag header found. The HTML meta robots tag is set to
  "index, follow" which is the default.
- **Note**: Not an issue -- the meta tag is sufficient.

### L2. Trailing Slash Handling
- **Status**: PASS
- **Evidence**: /leistungen and /leistungen/ both return 200 with the same content.
  No redirect between them.
- **Note**: While not generating redirect chains (good), the lack of redirect means
  both URL variants could be indexed. The canonical tag issue (C2) makes this moot
  since all canonicals point to homepage anyway.

### L3. No IndexNow Protocol
- **Status**: INFO
- **Evidence**: No IndexNow key file found. No IndexNow integration detected.
- **Impact**: New and updated content is not pushed to Bing/Yandex/Naver.
- **Recommendation**: Implement IndexNow for faster indexing on Bing and Yandex.
  Low effort, high value for content updates.

## PRIORITIZED REMEDIATION PLAN

### Priority 1: CRITICAL (Do immediately)

1. **Implement Server-Side Rendering or Static Site Generation**
   - Migrate from Vite + React CSR to Next.js (SSR/SSG) or Astro (SSG)
   - Vercel already hosts the site and natively supports Next.js
   - This single change resolves C1, C2, C3, H1, H2, and most other issues
   - Estimated effort: 2-4 weeks for a site of this size

2. **If SSR migration is not immediate, implement pre-rendering**
   - Use a pre-rendering service (Prerender.io, Rendertron) as interim solution
   - Configure Vercel middleware to serve pre-rendered HTML to crawlers
   - Ensure unique titles, descriptions, and canonicals per page

3. **Fix canonical tags NOW (even before SSR migration)**
   - Each page MUST have its own canonical URL
   - /leistungen/seo must canonical to https://webseite.hamburg/leistungen/seo
   - This can be done via JS (useEffect) as a temporary fix, though server-side is preferred

### Priority 2: HIGH (Within 2 weeks)

4. **Add og:image tags** -- Create branded OG images per page category
5. **Add Twitter Card meta tags** -- twitter:card, twitter:image
6. **Configure proper 404 responses** in vercel.json

## BOTTOM LINE

The site has excellent content, good URL architecture, forward-thinking AI readiness
(llms.txt), and rich structured data. However, the pure CSR architecture fundamentally
undermines all of these strengths because search engines may never see the content in
its intended form.

The #1 action item is migrating to server-side rendering. Given the site is already
on Vercel and built with React, migrating to Next.js is the natural path. This single
change would likely raise the technical SEO score from 38/100 to 75-85/100, with the
remaining points addressed by the medium and low priority items listed above.
